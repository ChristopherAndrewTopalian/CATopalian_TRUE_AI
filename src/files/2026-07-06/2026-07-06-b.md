Let's take these in order — the root question has a clean answer, and the spreadsheet/authoring question has a genuinely nice hybrid solution.

**How far back should "root" go?**

There's no single correct depth — it's a design choice, not a fact you're missing. The real question isn't "how far back does it truly go," it's "what depth is actually *useful* for the queries I'll ask this system." A chain like `robin → bird → animal → living thing → entity` is technically valid (this is exactly what formal ontologies like WordNet do — they go all the way up to something like "entity"), but going that far rarely helps *you*, because "entity" doesn't tell you anything new about a robin.

Practical rule: **stop climbing the root chain once going further stops adding useful, distinguishing information.** For your project, that's probably:

```
robin → bird → animal → (stop)
```

"Animal" is where you'd stop, not because it's philosophically the true bottom, but because "living thing" or "entity" beyond that doesn't help you answer any question you'd actually ask this system (nothing meaningfully different happens whether you know a robin is a "living thing" or not — you already know that once you know it's an animal). If later you find yourself wanting to distinguish animals from plants at some other point in the system, you add that level then — you're not locked in, since `root` is just a string you can always extend.

**How to actually type an object-of-objects (no `push`, since this is genuinely a different shape)**

You're right that this feels foreign coming from arrays — here's the direct mapping of what you already know to what's new:

```js
// what you're used to — array, you push new items on:
const cortexArray = [];
cortexArray.push({ concept: "bird" });

// object-of-objects — no push; you assign a new key instead:
const cortex = {};
cortex.bird = { root: null, related: ["robin", "eagle"], attributes: ["feathers"] };
cortex.robin = { root: "bird", related: ["worm"], colors: ["red breast"] };

// or, written as one literal up front, same as your array-of-objects style:
const cortex2 = {
  bird:  { root: null, related: ["robin", "eagle"], attributes: ["feathers"] },
  robin: { root: "bird", related: ["worm"], colors: ["red breast"] }
};
```

That's the whole trick — instead of `array.push(newThing)`, it's `object.newKey = newThing`. Same mental action ("add a new entry"), just addressed by name instead of appended to a list.

**On the redundancy problem — this is a real, solvable pain point, not just something to push through**

A small helper function removes a lot of the retyping, by giving each concept sensible defaults so you only type the fields that differ:

```js
function defineConcept(name, overrides = {}) {
  cortex[name] = {
    root: null,
    related: [],
    attributes: [],
    ...overrides
  };
}

defineConcept("bird", { related: ["robin", "eagle", "hawk"], attributes: ["feathers", "beak"] });
defineConcept("robin", { root: "bird", related: ["worm"], colors: ["red breast"] });
```

This alone should cut a good chunk of the boilerplate — you're not retyping `root: null` and empty arrays for every concept anymore.

**The spreadsheet question — here's the honest tradeoff**

Using a spreadsheet as the actual *runtime* structure is a bad idea, and your instinct to hesitate is correct: spreadsheets are fundamentally flat (rows and columns), so representing a nested array inside one cell means writing `["robin","eagle","hawk"]` as literal text in that cell — which means you'd need code to *parse* that text back into a real array every time you load it. You'd be doing extra work (serializing into a cell, then parsing back out) for something that's already natural in a JS file.

**But using a spreadsheet (or plain CSV) as your *authoring* format — separate from the runtime structure — is genuinely useful**, and solves the exact problem you described (comparing entries side-by-side, seeing what's the same/different across rows, working without two monitors). Here's the hybrid:

1. Author your data as a simple CSV, one row per concept, columns for each field, multi-value fields separated by semicolons:

```csv
concept,root,related,attributes
bird,,robin;eagle;hawk;crow,feathers;beak;hollow bones
robin,bird,worm;spring;morning,red breast;brown back
worm,,robin;dirt;rain,soft;blind
```

2. Write a small one-time script that reads that CSV and builds your real `cortex` object automatically:

```js
const fs = require('fs');

const rows = fs.readFileSync('concepts.csv', 'utf-8')
  .trim()
  .split('\n')
  .slice(1); // skip header row

const cortex = {};

for (const row of rows) {
  const [concept, root, related, attributes] = row.split(',');
  cortex[concept] = {
    root: root || null,
    related: related ? related.split(';') : [],
    attributes: attributes ? attributes.split(';') : []
  };
}

console.log(cortex);
```

This gets you the best of both: rows and columns for easy eyeballing/comparison while you're populating data (works in literally any spreadsheet tool, or even a plain text CSV file in VS Code, no LibreOffice jitter involved), and a clean object-of-objects as the actual thing your program runs on. You'd only run the conversion script once (or whenever you update the CSV), not on every program run — so it doesn't cost you anything at runtime.

**Quick note on the C++ comparison** — you're not wrong to find it heavier there; JavaScript's object literal syntax (`{ key: value }`) is genuinely one of the easiest ways to write this kind of nested, flexible data in any mainstream language, which is exactly why your "dynamic JSObject" C++ version from earlier needed so much more machinery (`variant`, `unordered_map`, runtime type checks) just to approximate what JS gives you for free. That's a real, accurate reason to prefer JS for this specific project, not just unfamiliarity with C++.
