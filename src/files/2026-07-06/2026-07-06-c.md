Keep it a single string — making it an array here would be solving a problem you don't actually have.

**The reason:** `robin.root = "bird"` is a pointer, not a copy. Once you have that pointer, you already have full access to everything bird knows about itself, aliases included — you just follow the pointer:

```js
console.log(cortex[cortex.robin.root].aka); // → ["bird", "birdie", "avian"]
```

You never need to duplicate bird's aliases onto robin, because robin doesn't need to *carry* that information — it just needs to know *where to look for it*. That's the whole benefit of using references instead of copies: one source of truth, and everything downstream just points at it.

**When would `root` actually need to be an array?** Only when a concept genuinely has more than one parent category — true multiple inheritance, not just multiple names for the same parent. For example:

```js
cortex.bat = {
  root: ["mammal", "flying_creature"], // genuinely two different parent categories
  related: ["cave", "night", "echolocation"]
};

cortex.penguin = {
  root: ["bird", "swimmer"], // bird by biology, swimmer by behavior/category
  related: ["ice", "ocean", "flightless"]
};
```

That's a real, legitimate use for an array — a bat isn't "a kind of mammal that also happens to have an alias called flying_creature," it genuinely inherits traits from two separate concept branches at once. That's a different situation from "bird has three names for itself," which is just aliasing, not multiple parentage.

So the test to apply going forward: **if you're tempted to make a field an array, ask whether you're storing multiple *distinct things* (real multi-parent case → array) or just multiple *names for the same one thing* (aliasing case → keep it singular, and store the names once, on the concept itself, then reference it).** Robin only has one biological parent category (bird), so singular is correct here — you'd only reach for the array form the day you add something like a bat or a penguin that legitimately branches two ways.
